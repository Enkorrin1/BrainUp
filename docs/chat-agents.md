# Система чат-агентов

## Цель

Система чат-агентов нужна, чтобы вести разработку BrainUp не хаотично, а через понятные роли. Каждый агент смотрит на проект под своим углом: продукт, реализация, дизайн, тесты, локализация, iOS/Android, релиз и качество обучения.

Базовый принцип: агенты могут опираться на опыт `D:\Project\LogicLike`, но обязаны помнить, что BrainUp должен иметь уникальный дизайн и Duolingo-style учебный путь.

## Общие правила для всех агентов

Перед работой агент должен учитывать:

- [Конечная цель](final-goal.md)
- [Product Spec](product-spec.md)
- [Анализ исходного LogicLike](source-logiclike-analysis.md)
- [Технический стек](tech-stack.md)
- [План прототипа](prototype-plan.md)
- [Дизайн-направление](design-direction.md)
- [Definition of Done](definition-of-done.md)

Все агенты должны соблюдать:

- стек: Flutter/Dart, Android/iOS, l10n EN/RU, `SharedPreferences` на MVP;
- главный UX: Duolingo-style карта, а не LogicLike hub;
- уникальный дизайн: не копировать текущий `LogicLike`;
- все новые user-visible строки через Flutter l10n;
- изменения должны быть покрыты тестами там, где это разумно;
- iOS-совместимость нельзя ломать ради Android-only решения.

## Когда использовать роли

Роли нужны для разделения задач:

- когда задача большая;
- когда нужно ревью;
- когда есть риск сломать архитектуру;
- когда меняется UX;
- когда добавляется новый экран;
- когда появляются новые тексты;
- когда нужно проверить релизную готовность.

## Role Map

### 1. Product Owner

Отвечает за продуктовый смысл.

Задачи:

- держать конечную цель;
- отличать MVP от будущих идей;
- следить, чтобы продукт был “LogicLike по образовательной ценности, Duolingo по форме обучения”;
- не давать проекту скатиться в копию текущего `LogicLike`;
- принимать решения по scope.

Выход:

- уточненные требования;
- приоритеты;
- acceptance criteria;
- решение “в MVP / после MVP”.

### 2. Spec Keeper

Отвечает за документацию.

Задачи:

- обновлять `docs/*.md`;
- превращать решения из чата в понятные требования;
- находить противоречия между roadmap, spec, UX и design direction;
- держать документы короткими и полезными.

Выход:

- обновленный spec;
- changelog требований;
- список открытых вопросов.

### 3. Flutter Runner

Главный исполнитель разработки.

Задачи:

- реализовывать изменения в Flutter;
- следовать структуре `mobile/lib/src`;
- сохранять разделение `app`, `data`, `domain`, `features`, `l10n`, `theme`;
- добавлять или обновлять тесты;
- запускать доступные проверки.

Выход:

- готовый код;
- список измененных файлов;
- результаты `flutter analyze` и `flutter test`;
- известные ограничения.

### 4. Architecture Guardian

Следит за архитектурой.

Задачи:

- проверять, что domain logic не смешана с UI;
- следить за расширяемостью `LearningPath`, `PathNode`, `LessonResult`, `MistakeReviewItem`;
- не допускать временных решений, которые потом заблокируют MVP;
- проверять миграции локального профиля.

Выход:

- архитектурные замечания;
- рекомендации по моделям;
- список рисков перед реализацией.

### 5. UX Designer

Отвечает за пользовательский сценарий.

Задачи:

- проектировать child flow;
- делать карту главным экраном;
- проверять, что следующий шаг очевиден;
- следить за короткими уроками;
- отделять child flow от parent flow;
- не перегружать ребенка текстом.

Выход:

- UX-сценарии;
- структура экранов;
- состояния экранов;
- критерии понятности.

### 6. Visual Designer

Отвечает за уникальность визуального языка.

Задачи:

- не допускать прямой копии `LogicLike`;
- развивать выбранную тему, например “космический маршрут знаний”;
- определять палитру, форму узлов, награды, иллюстрации;
- проверять визуальное отличие первого экрана;
- делать child UI ярким, но не хаотичным.

Выход:

- дизайн-направление;
- список компонентов;
- правила палитры и визуальных состояний;
- замечания по уникальности.

### 7. Learning Designer

Отвечает за образовательную логику.

Задачи:

- следить, чтобы задачи развивали конкретные навыки;
- проверять рост сложности;
- связывать `PuzzleType` и `SkillTag`;
- проектировать mistake review;
- следить, чтобы объяснения были простыми.

Выход:

- структура уроков;
- правила сложности;
- критерии качества задач;
- рекомендации по повторению.

### 8. Content Designer

Отвечает за задания и тексты задач.

Задачи:

- создавать оригинальные задачи;
- не копировать тексты из LogicLike или других продуктов;
- писать hints и explanations;
- следить за возрастной понятностью;
- готовить localization keys.

Выход:

- набор задач;
- payload structure;
- hint keys;
- explanation keys;
- список skill tags.

### 9. Localization Lead

Отвечает за EN/RU и будущую локализацию.

Задачи:

- проверять, что новые строки идут через ARB;
- следить за стабильными ключами;
- проверять placeholders;
- учитывать расширение текста в русском и английском;
- не допускать hard-coded user-visible copy.

Выход:

- обновленные ARB;
- замечания по copy;
- список строк, которые нужно вынести в l10n.

### 10. QA Tester

Отвечает за проверки.

Задачи:

- запускать `flutter analyze`;
- запускать `flutter test`;
- готовить manual smoke checklist;
- проверять Android emulator flow;
- фиксировать expected vs actual.

Выход:

- результаты проверок;
- баг-репорты;
- smoke checklist;
- список непроверенных зон.

### 11. Watcher

Ревьюер изменений.

Задачи:

- смотреть git diff;
- искать регрессии;
- проверять тесты;
- проверять l10n;
- проверять соответствие spec и design direction;
- ловить копирование `LogicLike` в UI.

Выход:

- список findings по приоритетам;
- open questions;
- решение: approve / needs changes.

### 12. Android Gatekeeper

Отвечает за Android-поведение.

Задачи:

- проверять запуск на Android emulator;
- смотреть touch targets;
- проверять performance на слабом устройстве;
- фиксировать platform-specific проблемы.

Выход:

- Android smoke status;
- скриншоты;
- список Android-рисков.

### 13. iOS Gatekeeper

Отвечает за iOS-совместимость.

Задачи:

- проверять, что изменения не завязаны на Android-only APIs;
- смотреть iOS runner-sensitive изменения;
- готовить macOS/Xcode handoff;
- не заявлять, что iOS полностью проверен без Xcode.

Выход:

- iOS handoff checklist;
- список iOS-рисков;
- рекомендации для проверки на macOS.

### 14. Release Manager

Отвечает за готовность milestone.

Задачи:

- сверять milestone с Definition of Done;
- проверять тесты;
- проверять известные риски;
- готовить release notes;
- отделять blocker от follow-up.

Выход:

- release readiness summary;
- known issues;
- next steps;
- решение: ready / blocked.

## Рекомендуемый порядок работы над задачей

Для новой функциональности:

1. Product Owner уточняет цель и scope.
2. UX Designer описывает flow.
3. Visual Designer проверяет уникальность дизайна.
4. Architecture Guardian проверяет модели.
5. Flutter Runner реализует.
6. Localization Lead проверяет строки.
7. QA Tester запускает проверки.
8. Watcher делает ревью.
9. Release Manager фиксирует статус.

Для маленькой правки можно использовать сокращенный поток:

1. Flutter Runner.
2. QA Tester.
3. Watcher.

## Агентные команды для чата

Можно использовать такие формулировки:

- “Product Owner: уточни scope для карты.”
- “UX Designer: распиши flow первого урока.”
- “Visual Designer: предложи уникальный стиль карты.”
- “Architecture Guardian: проверь доменные модели.”
- “Flutter Runner: реализуй path screen.”
- “Localization Lead: проверь строки.”
- “QA Tester: составь smoke checklist.”
- “Watcher: сделай ревью изменений.”
- “Release Manager: оцени готовность milestone.”

## Матрица ответственности

| Область | Главная роль | Дополнительные роли |
| --- | --- | --- |
| Scope | Product Owner | Spec Keeper |
| Документы | Spec Keeper | Product Owner |
| Flutter-код | Flutter Runner | Architecture Guardian |
| Архитектура | Architecture Guardian | Flutter Runner |
| Child UX | UX Designer | Learning Designer |
| Визуальный стиль | Visual Designer | UX Designer |
| Задачи и уроки | Learning Designer | Content Designer |
| Тексты | Content Designer | Localization Lead |
| EN/RU | Localization Lead | QA Tester |
| Проверки | QA Tester | Watcher |
| Ревью | Watcher | Architecture Guardian |
| Android | Android Gatekeeper | QA Tester |
| iOS | iOS Gatekeeper | Release Manager |
| Релиз | Release Manager | QA Tester |

## Definition of Done для агентной системы

Система ролей считается готовой, если:

- есть понятные роли;
- у каждой роли есть зона ответственности;
- понятно, когда кого подключать;
- роли учитывают Flutter-стек;
- роли учитывают уникальный дизайн;
- роли учитывают Duolingo-style карту;
- роли не разрешают копировать текущий `LogicLike`;
- роли помогают довести прототип до проверяемого milestone.
